## Highlights

- **Rich markdown notes.** Credential notes now render as formatted markdown
  through a curated GitHub-Flavoured subset: ATX and Setext headings,
  ordered and unordered lists (including nested), block quotes, fenced and
  indented code blocks, GFM tables, strikethrough, autolinks, and inline
  emphasis / strong / code. Source-mode and preview-mode toggle from the
  credential detail screen. Live edit preview reuses the same renderer with
  an offset-mapped cursor so caret position survives the round-trip.
- **Autofill biometric and PIN unlock.** The autofill picker can now unlock
  straight from biometric or PIN without bouncing through the main activity.
  Aborted autofill sessions clear the capture buffer so a forgotten dialog
  never leaks fields across credentials.
- **Lock-screen polish.** The PIN field keeps focus during the Loading state
  so the IME survives a wrong-PIN bounce; biometric auto-trigger is gated on
  window focus so the prompt no longer flickers when the screen turns off
  mid-prompt; the lock screen is portrait-locked while mounted.

## Security and privacy posture

The new markdown surface is by far the largest new attack surface in this
release. The design brief was explicit: a malicious imported note must not be
able to escalate beyond visible glyphs. The defences are summarised below.
The full threat model is documented further down in this file under
**Markdown notes - threat model addendum** (this content would normally live
in a repo-root `CLAUDE.md`, but no `CLAUDE.md` exists in the repo yet, so the
addendum rides along here for the v1.1.0 release).

- **XSS-safe rendering pipeline.** Inline HTML tags (for example
  `<script>...`, `<img onerror=...>`) flow through the renderer as literal
  glyphs. Compose `Text` does not interpret tag syntax, so a malicious
  imported note cannot escalate to script execution. The visible bytes equal
  the source bytes for any HTML-shaped fragment.
- **No image loading, by design.** `IMAGE` AST nodes render as the literal
  text `[image: <alt>]`. No image loader (Coil, Glide, Picasso, OkHttp,
  Compose `AsyncImage`) is referenced anywhere in the markdown package.
  The URL is never opened. The pixel-loading attack surface is zero.
- **Scheme allowlist on every link tap.** Tap dispatch routes through
  `SchemeAllowlistUriHandler`. ALLOW: `http`, `https`, `mailto`. DENY
  (silently, surfaced via `onBlockedLink` for a snackbar): everything else,
  including `javascript:`, `intent:`, `content:`, `file:`, `data:`, custom
  app schemes, `tel:`, `sms:`. Scheme policy is enforced at click-dispatch
  time, not at parse time, so changing the allowlist later does not require
  touching the renderer.
- **Confirm-before-open with homograph warnings.** Every link tap is gated
  by `LinkConfirmDialog`. The dialog renders the full destination URL
  (no ellipsis, `softWrap = false` with horizontal scroll) and shows a
  "This domain looks similar..." warning row when `HomographDetector` flags
  a punycode host or a digit-letter confusable (`paypa1.com` trips the
  `1`-for-`l` confusable layer).
- **64 KiB size cap on the parser.** Notes larger than 64 KiB UTF-8
  short-circuit to a verbatim plain-text fallback BEFORE the JetBrains
  parser is invoked. The host screen shows a non-dismissible banner
  (`FallbackReason.SIZE_CAP_EXCEEDED`) when the cap is hit. Any parser
  throwable is also caught (`FallbackReason.PARSE_FAILED`) and the source
  renders verbatim, so a pathological grammar fragment cannot ReDoS the UI.
- **Pinned, vetted parser dependency.** `org.jetbrains:markdown` is pinned
  to **0.7.3** in `gradle/libs.versions.toml` (Apache-2.0). This library
  is on the security-critical path; bumps must go through a manual review
  (see the addendum below).
- **Adversarial corpus tests.** New unit tests under
  `app/src/test/java/com/onekey/core/presentation/markdown/` exercise the
  renderer against deeply nested inline emphasis, table rows that hit GFM
  ambiguity edges, mixed BMP / non-BMP code points, surrogate pair
  boundaries, autolinks that re-enter code spans, and the >64 KiB
  short-circuit branch.
- **Notes normalised at every chokepoint.** `CredentialRepositoryImpl` and
  `VaultImporterImpl` now run notes through `NotesNormalize` before
  persisting, so a malicious export from another vendor cannot smuggle
  CRLF, NUL, or zero-width sequences into the vault.
- **Settings toggle.** A global "Render markdown in notes" preference lives
  under Settings -> General -> Notes (default ON). Off -> every note
  renders as plain text regardless of source.

## What else changed

- **Settings: autofill category-filter.** Filter which credential categories
  the autofill picker offers.
- **Schema v15.** Adds an `importedAt` per-credential timestamp for the
  import surfaces. Migration tested via Room `MigrationTestHelper`.
- **Theme picker.** Replaced the binary dark-mode toggle with a
  Follow system / Light / Dark picker (Follow system is the new default).
- **APK split by ABI.** Release builds now produce `arm64-v8a`,
  `armeabi-v7a`, `x86_64`, plus a universal fallback. APK names follow
  `1Key_<versionName>_<versionCode>_<abi>_<buildType>.apk`.
- **Compose BOM 2025.08.00.** Brings high-refresh-rate scrolling fixes and
  the Kotlin 1.9 compatibility patches.
- **TopAppBar pinned to surfaceContainer.** Matches the bottom-nav chrome,
  shrunk to 56 dp via `expandedHeight` and a pre-clamped scroll state.
- **Cold-start tuning.** `TagRepository` is now deferred from
  `Application.onCreate`. Other singletons stay eager; the rationale is
  documented in code.
- **Root detection off the main thread.** `RootDetector.check()` runs in a
  `LaunchedEffect` on `Dispatchers.IO`.
- **Repo-wide character sweep.** Replaced non-ASCII arrows and en-dashes in
  source comments and a handful of Settings guide strings with ASCII
  equivalents. User-facing typography on the password-generator labels
  (`Uppercase (A-Z)` and the like) is intentionally preserved.

## Known limits

- **TalkBack manual walk deferred.** The new `MarkdownNotesAccessibilityTest`
  covers heading / link / list / selection cases under Compose UI testing,
  but a real-device TalkBack walk against a 64 KiB stress note is deferred
  to a v1.1.x patch.
- **Per-character link contentDescription not yet split.** Today the
  renderer attaches one `contentDescription` per link-bearing block.
  A future change can split each link into its own sub-`Text` so TalkBack
  reads `Link: <host>` per anchor instead of a concatenated list.
- **No image rendering, by design.** Image syntax parses, but renders as a
  placeholder. A future flag could allow trusted-host image loading; today
  the safest stance is to ship zero pixels.
- **Dependency-verification metadata not yet generated.** Gradle's
  `verification-metadata.xml` is deferred to a future release (see the
  addendum below).
- **Autofill: inline IME suggestions and passkey support not in this
  release.** Path A shipped; inline IME and passkeys remain on the roadmap.
- **No auto-clear of the rendered preview on host-activity background.**
  When you background the app, the rendered note remains in the composition
  until the vault locks. The auto-lock timer is the bound; the renderer does
  not separately scrub the rendered output.
- **No per-code-block copy button.** Fenced code blocks render with
  monospaced typography but do not get a "copy" affordance. Use the system
  long-press selection to copy code-block content.
- **No syntax highlighting inside fenced code blocks.** Code fences render
  in monospace with a tinted background; language tags (e.g. ```` ```kotlin ````)
  are parsed but not coloured. Lexer-based highlighting is a separate
  initiative and not in v1.1.0.

## The seven cursor decisions

The live preview editor (Flavour 2: hide balanced markers) is a small UX
language: it lets the user type CommonMark + GFM and have the rendering
fold immediately, while the source is what gets saved. The seven cursor
decisions below were all locked in product review BEFORE coding started
and are encoded directly in `MarkdownEditorField` + `MarkdownEditorOffsetMapping`.

1. **Backspace at the visible end of a balanced marker pair deletes the
   entire closing marker run as one unit.** Backspacing at the visible
   end of `**bold**` removes the closing `**` in a single keypress and
   the markers reappear as `**bold` (visually, with the asterisks now
   re-shown). Behaviour proven by `backspaceAtEndOfBoldDeletesMarkerPair`.
2. **Tap inside a balanced span lands the cursor at the correct
   underlying offset.** The offset mapping snaps to the original
   character position even when the visible text and underlying source
   differ in length (e.g. `[image: alt]` placeholder for `![alt](url)`).
3. **Word-selection copies the VISIBLE rendered text only.** Long-press
   selecting `bold` inside `**bold**` and tapping Copy puts `bold` on
   the clipboard, not `**bold**`. Achieved via `MarkdownAwareTextToolbar`
   which intercepts the copy/cut request and substitutes the transformed
   slice.
4. **Full-line copy yields visible text.** Same toolbar override handles
   the multi-line case; copying a heading row yields `Title` rather than
   `# Title`.
5. **Arrow keys jump past hidden marker regions in one keypress.** Right
   arrow at the visible end of `**bold**` lands the cursor past the
   closing `**` in a single press. Same for Left arrow at the visible
   start past the opening `**`. Symmetric at the source buffer edge so a
   standalone `**bold**` (no trailing visible glyph) does not require a
   double keypress to escape. Proven by
   `rightArrowJumpsPastHiddenMarkers` and `leftArrowJumpsPastHiddenMarkers`.
6. **Typing at the marker boundary stays INSIDE the formatting.** Cursor
   at the visible end of `**bold**`, typing `s`, yields `**bolds**`
   (visible `bolds`, still bold) rather than `**bold**s` (visible
   `bolds`, the `s` outside the format). Press Right once first to
   escape the formatting before typing if you want the new character
   unformatted.
7. **Paste re-parses live.** Pasted markdown is parsed by the same path
   as typed input; balanced markers fold immediately, unbalanced markers
   show verbatim.

The 64 KiB cap is enforced at `onValueChange` and rejects the whole
paste wholesale (no silent truncation) so an oversized clipboard payload
cannot land in a half-formed state.

# Markdown notes - threat model addendum

> This section would normally live in a repo-root `CLAUDE.md` alongside
> `CONTRIBUTING.md` / `PRIVACY.md`. No `CLAUDE.md` exists today, so the
> content rides along here. Lift this whole section into a future
> `CLAUDE.md` when the file is introduced.

The credential-detail "Notes" surface renders a curated GFM subset through
`app/src/main/java/com/onekey/core/presentation/markdown/`. The pipeline is
deliberately small and the safety claims are precise.

## Scheme allowlist

- Every link tap routes through `SchemeAllowlistUriHandler.kt`.
- **ALLOW:** `http`, `https`, `mailto`.
- **DENY** (silently, surfaced through `onBlockedLink` for a snackbar):
  every other scheme, including `javascript:`, `intent:`, `content:`,
  `file:`, `data:`, `tel:`, `sms:`, `ftp:`, `vbscript:`, `ws:`, `wss:`,
  and any unknown / app-custom scheme.
- The renderer captures URLs as `URL_ANNOTATION_TAG` annotations regardless
  of scheme. Scheme policy is enforced at click-dispatch time, NOT at parse
  time, so widening or narrowing the allowlist later does not require a
  renderer change.

## Image rendering

- `IMAGE` AST nodes render as the literal text `[image: <alt>]` and never
  load any URL.
- No image loader (Coil, Glide, Picasso, OkHttp, Compose `AsyncImage`,
  `rememberAsyncImagePainter`) is referenced by the markdown package.
  Verified by repo grep across
  `app/src/main/java/com/onekey/core/presentation/markdown/`.
- Do NOT add an image loader to this package without a separate threat
  model write-up. Pixel-loading is the largest attack surface a markdown
  renderer can grow; the project's no-INTERNET stance makes it gratuitous.

## Inline HTML

- `HTML_BLOCK` and inline `HTML_TAG` tokens flow through as glyph streams
  via `Text`. Compose `Text` does NOT interpret tag syntax, so `<script>`,
  `<img onerror=...>`, and other XSS payloads are structurally inert.
- The visible bytes equal the source bytes. This is intentional: a user
  who wrote `<b>foo</b>` should see `<b>foo</b>` and be able to copy that
  string out untouched.

## Size cap (ReDoS mitigation)

- `MAX_SOURCE_BYTES = 64 * 1024` in `MarkdownNotesView.kt` is the hard cap.
- Sources larger than 64 KiB UTF-8 short-circuit to a plain-text fallback
  BEFORE the JetBrains parser is invoked. The host screen shows a
  non-dismissible banner (`FallbackReason.SIZE_CAP_EXCEEDED`).
- Any throwable from the parser is also caught
  (`FallbackReason.PARSE_FAILED`) and the source renders verbatim.
- Adversarial test corpus lives under
  `app/src/test/java/com/onekey/core/presentation/markdown/`. New
  adversarial cases (deep nesting, table ambiguity, surrogate pair
  boundaries, autolink re-entry into code spans) must land alongside any
  expansion of the supported AST set.

## Library pin and update policy

- Coordinate: `org.jetbrains:markdown` (KMP umbrella; resolves to
  `markdown-jvm-0.7.3` on Android).
- Pinned version: **0.7.3** at
  `gradle/libs.versions.toml` (`jetbrainsMarkdown = "0.7.3"`).
- License: Apache-2.0.
- **Manual-review rule.** This library is on the security-critical path.
  Renovate / Dependabot / any auto-bump tool MUST NOT merge a major or
  minor bump to `org.jetbrains:markdown` automatically. Required steps
  before any version bump:
  1. Diff the upstream `CHANGELOG.md` between the current and proposed
     versions; call out any new AST node type or parser flag.
  2. Re-run the adversarial corpus tests. Add a new case for any newly
     supported AST construct.
  3. Re-verify the `[image: alt]` placeholder and `HTML_TAG` glyph-stream
     behaviour against the new parser output. These are the two surfaces
     XSS regressions would land in.
  4. Re-record the 32 KiB and 64 KiB parse-time benchmarks (Pixel 6, M3
     Compose); confirm the cap rationale comment in `MarkdownNotesView.kt`
     still holds.
- Patch bumps (for example 0.7.3 -> 0.7.4) MAY auto-merge if the
  changelog explicitly states no parser or AST changes.
- If a Renovate config lands later, encode the rule under `packageRules`
  with `"automerge": false` and a `security-review-required` label for
  `org.jetbrains:markdown`.

## Dependency-verification metadata - deferral note

- Gradle's `verification-metadata.xml`
  (`./gradlew --write-verification-metadata sha256`) is NOT yet generated.
- Rationale: the project has no INTERNET permission and ships no SaaS
  surface, so a supply-chain compromise affects build-time integrity only.
  The risk surface is the developer machine, not user devices. We chose to
  defer until the dependency graph stabilises.
- Re-evaluate at v1.2, or as soon as any networked dependency is added
  (currently zero).
- Do NOT cite "verification metadata" as a shipped guarantee in any
  release note or whitepaper until the file exists in the repo.

## Commit and style conventions for this surface

- One-line subject; no `Co-Authored-By` trailer; never stage `.claude/`
  or any agent-tooling files.
- ASCII-only source comments. No `->` arrow glyph (U+2192), no en-dash
  (U+2013), no em-dash (U+2014). User-facing UI typography MAY use them
  when the visual character is intentional (for example the
  password-generator character-range labels).
- Avoid the section-sign character. Spell out "section" instead.
